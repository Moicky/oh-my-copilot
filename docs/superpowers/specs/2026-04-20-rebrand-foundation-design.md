# Sub-project #1 — Rebrand & Packaging Foundation

**Status:** Approved design (pending user spec review)
**Parent program:** Full functional port of `oh-my-codex` (OMX) to GitHub Copilot CLI as `oh-my-copilot` (OMCP). This is sub-project 1 of ~7.
**Scope of this sub-project:** Identity-only. After it lands, the project is named, packaged, branded, and released as `oh-my-copilot`, but it does **not yet** function against Copilot CLI. Runtime swap, hook redesign, skills migration, etc. are later sub-projects.

---

## Problem

The repository is a fork of `Yeachan-Heo/oh-my-codex` (OMX), an extensive workflow layer for the OpenAI Codex CLI. The user wants the fork to become `oh-my-copilot` (OMCP), targeting the GitHub Copilot CLI. A full port spans ~7 sub-projects; this sub-project establishes the new identity so all later work has a clean naming foundation.

The challenge: ~308 files mention "codex", but those references fall into two categories:

- **Brand identity** — names, packaging, docs, file/dir names, env var prefixes scoped to OMX itself. **In scope here.**
- **Functional integration** — calls to the `codex` CLI, references to `~/.codex/` config, `CODEX_HOME`, native Codex hook formats. **Out of scope here**, owned by sub-projects #2/#3.

Conflating the two would either leak runtime work into this sub-project or leave broken half-renames. We keep them strictly separated.

## Approach

A single mechanical rename pass plus targeted document rewrites and deletions, applied in five ordered phases. Each phase leaves the repository in a buildable-or-clearly-broken state and is independently committable.

The rename rules use **exact-token** replacements (word-boundary aware) to avoid touching the external `codex` CLI surface:

- `oh-my-codex` → `oh-my-copilot`
- `\bOMX\b` → `OMCP`
- `\bomx\b` → `omcp` (binary, command, identifier)
- `omx-` (Cargo crate prefix) → `omcp-`
- `\.omx/` and `\.omx\b` (state dir) → `.omcp/` / `.omcp`
- `Yeachan-Heo/oh-my-codex` → `Moicky/oh-my-copilot`

**Not** in the rename rules (intentionally preserved):

- `codex` (the external CLI binary name)
- `\.codex/` (the upstream Codex CLI config directory)
- `CODEX_HOME` (upstream env var)
- `@openai/codex` (upstream npm package install instructions — these will be removed/replaced in sub-project #2 when we migrate to `@github/copilot`)
- The word "codex" inside historical/deleted files (those files are deleted, not rewritten)

## Phases

### Phase A — Identity files & manifests

Touch only the small set of files that define the project's identity to npm, cargo, and tooling.

- `package.json`
  - `name` → `@moicky/oh-my-copilot`
  - `version` → `0.1.0`
  - `description` → "Workflow layer for GitHub Copilot CLI (forked from oh-my-codex)"
  - `bin.omx` → `bin.omcp`, path `dist/cli/omcp.js`
  - `repository` → `Moicky/oh-my-copilot`
  - `scripts` referencing `dist/cli/omx.js` → `dist/cli/omcp.js`
  - All script invocations of the local binary (`node dist/cli/omx.js …`) → `node dist/cli/omcp.js …`
  - `lint` paths if necessary
- `Cargo.toml` (workspace root)
  - `members` paths → `crates/omcp-*`
  - `workspace.package.repository` → `https://github.com/Moicky/oh-my-copilot`
  - `workspace.package.version` → `0.1.0`
- Each `crates/omx-*/Cargo.toml`
  - `package.name` → `omcp-*`
  - Inter-crate `dependencies.omx-*` → `dependencies.omcp-*`
- `dist-workspace.toml` — version/repository references
- `biome.json`, `tsconfig.json`, `tsconfig.no-unused.json` — only if they reference renamed paths
- `Cargo.lock` — regenerated by `cargo build` / `cargo metadata` after crate rename; commit the regenerated file

### Phase B — Directory & file renames (git mv)

- `crates/omx-explore` → `crates/omcp-explore`
- `crates/omx-mux` → `crates/omcp-mux`
- `crates/omx-runtime` → `crates/omcp-runtime`
- `crates/omx-runtime-core` → `crates/omcp-runtime-core`
- `crates/omx-sparkshell` → `crates/omcp-sparkshell`
- `src/cli/omx.ts` → `src/cli/omcp.ts` (and any other `omx.*` source files surfaced by the audit step)
- Any other file/dir whose name contains `omx` (audit via `find . -name '*omx*' -not -path './node_modules/*' -not -path './target/*' -not -path './dist/*'`)

Use `git mv` to preserve history.

### Phase C — Source-token rename (mechanical, scoped)

Run the exact-token replacements listed in **Approach** across all tracked source files, **excluding**:

- `node_modules/`, `dist/`, `target/`, `Cargo.lock` (regenerated separately), `package-lock.json` (regenerated separately)
- The historical docs slated for deletion in Phase E (avoid wasted churn)
- The non-English README translations slated for deletion in Phase E

Verification after this phase:

- `rg '\bOMX\b|\bomx\b|oh-my-codex' -g '!node_modules' -g '!dist' -g '!target' -g '!Cargo.lock' -g '!package-lock.json'` returns **zero hits** outside intentionally preserved spots (e.g. fork-attribution line, deleted-doc remnants if any survived).
- `rg '\bcodex\b'` still returns hits — those are the functional references owned by sub-project #2/#3. This is expected.
- `npm install` and `npm run build` still succeed (the rename is purely identity; logic is untouched).
- `cargo build` succeeds for all renamed crates.

### Phase D — Documentation rewrite

- **README.md** — full rewrite around the new identity. Sections:
  - Title `# oh-my-copilot (OMCP)`
  - Brief tagline
  - One-line attribution: `> Forked from [oh-my-codex](https://github.com/Yeachan-Heo/oh-my-codex) by Yeachan Heo. Re-targeted at GitHub Copilot CLI.`
  - Status banner: "🚧 Early fork — runtime is being ported from Codex CLI to Copilot CLI. Most commands do not work yet. See ROADMAP."
  - Roadmap pointer to `docs/superpowers/specs/`
  - License (MIT) + attribution to upstream
  - **Removed:** Discord/community section, maintainer table, ambassador table, top-collaborators table, star-history badge, character image, language switcher, all "Recommended default flow" sections that depend on Codex (replaced by the status banner)
- **CHANGELOG.md** — replace contents with a single entry:
  ```
  # Changelog

  ## 0.1.0 — 2026-04-20

  Initial fork from oh-my-codex 0.14.0. Renamed to oh-my-copilot (OMCP).
  Rebrand and packaging only — runtime is not yet ported to Copilot CLI.
  ```
- **CONTRIBUTING.md** — rebrand mechanically (Phase C catches most), then strip references to removed community channels.
- **DEMO.md, COVERAGE.md** — Phase C handles tokens. Manually inspect for any community/maintainer mentions and remove.
- **RELEASE_BODY.md** — replace with a fork-launch body:
  ```
  oh-my-copilot 0.1.0 — Initial fork

  Renamed and rebranded from oh-my-codex 0.14.0. Runtime port to GitHub
  Copilot CLI is in progress; this release is identity-only and not
  functional. See README for status.
  ```
- **docs/_config.yml, docs/index.html, docs/getting-started.html, docs/agents.html, docs/skills.html, docs/integrations.html** — Phase C handles tokens; manually remove references to deleted historical docs.

### Phase E — Deletions

- **All `docs/readme/README.*.md`** (15 non-English translations).
- **Historical docs**:
  - `docs/release-notes-*.md`
  - `docs/release-body-*.md`
  - `docs/qa-plan-*.md`, `docs/qa-report-*.md`
  - `docs/migration-*.md`
  - `docs/prs/`, `docs/issues/`
  - `docs/prompt-migration-changelog.md`
- **External branding** — strip image/badge URLs pointing at `yeachan-heo.github.io` and `api.star-history.com/svg?repos=Yeachan-Heo/oh-my-codex` from any remaining doc.
- **`.github/`**:
  - `ISSUE_TEMPLATE/*` — rebrand or replace with a single minimal `bug.md` and `feature.md` template. Remove links to original Discord.
  - `PULL_REQUEST_TEMPLATE.md` — strip references to upstream-specific processes.
  - `workflows/release.yml`, `workflows/ci.yml`, `workflows/pr-check.yml`, `workflows/dev-merge-issue-close.yml` — Phase C catches tokens. Additionally:
    - Disable npm publish step or change registry target to `@moicky` scope.
    - Update cargo-dist release names if hardcoded.
    - Add a comment block at the top: "Workflows inherited from oh-my-codex; functional behavior re-validated in later sub-projects."
- Drop `docs/openclaw-integration.*.md` non-English variants (keep `openclaw-integration.md` for now; OpenClaw subsystem is owned by sub-project #7).

### Phase F — Verification & sanity

- `npm install && npm run build` succeeds.
- `cargo build --workspace` succeeds.
- `cargo test --workspace --no-run` compiles (running tests not required — many will fail because Codex isn't being driven; that's later).
- `node dist/cli/omcp.js --help` runs and prints help with no `omx`/`OMX` strings in the output.
- Final grep audit:
  ```
  rg -i 'oh-my-codex|\bomx\b|\bOMX\b|yeachan-heo' \
     -g '!node_modules' -g '!dist' -g '!target' \
     -g '!Cargo.lock' -g '!package-lock.json'
  ```
  Only allowed hits: the single fork-attribution line in README.md and the upstream link in CHANGELOG/RELEASE_BODY if kept.

## Risks & mitigations

| Risk | Mitigation |
| --- | --- |
| Mechanical rename hits a string we wanted to preserve (e.g. inside a quoted log line referring to upstream Codex behavior) | Word-boundary regex + post-rename grep audit + reviewer eye on Phase F output |
| `Cargo.lock` regeneration produces unexpected diff | Run `cargo update -p <renamed crate>` only on renamed crates; commit lock as a separate logical commit |
| CI workflows reference removed files (e.g. release notes, language READMEs) | Phase E updates workflows in same commit as deletions |
| Source identifier collision: a variable literally named `omx` in code might be ambiguous with the brand | Code identifiers in this codebase use `omx` as the brand prefix — renaming all of them is correct. No collisions expected. |
| `package-lock.json` churn from name change | Regenerate with `npm install`, commit as own logical commit |
| Sub-project #2 starts before #1 is merged and conflicts | Sequence sub-projects strictly; #2 design starts only after #1 ships |

## Deliverables

- Repository builds (`npm run build`, `cargo build --workspace`).
- `omcp --help` (or equivalent help-only command path that doesn't touch the missing Codex runtime) runs cleanly.
- README, CHANGELOG, RELEASE_BODY reflect the fork.
- All audit greps pass.
- A single commit (or small ordered series) per phase, all on a feature branch, ready for sub-project #2 to build on.
